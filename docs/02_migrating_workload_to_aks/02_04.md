# Task 04 - Creating the AKS Cluster

## Introduction

In this task, you will be able to create the AKS cluster in your resource group and connect `kubectl` to AKS

## Description

After you complete this exercise, you will be able to: 

- Create and validate the creation of an AKS cluster 

- Connect Kubectl to your AKS cluster 

## Success Criteria

* Successfully generate the resources of the AKS cluster in your subscription
* Connect Kubectl to your new created AKS cluster 

## Solution

<details markdown="block">
<summary>Expand this section to view the solution</summary>

1. You will now create an AKS cluster in your resource group already established.  From your termnial, issue the following az command:

    ```bash
    az aks create \
      --resource-group rg-tailspin \
      --name tailspin_aks1000 \
      --node-count 1 \
      --enable-addons monitoring \
      --generate-ssh-keys \
      --attach-acr tailspinacr1000
    ```

> **Note:** This command will take several minutes to complete.

You can verify in the portal that your resource group now contains resources for 

2. Get the credentials for AKS using the following command:

    ```bash
    az aks get-credentials \
      --resource-group rg-tailspin \
      --name tailspin_aks1000
    ```

This will save the credentials locally on your machine in your user's folder under `.kube\config`

3. Execute the command to get nodes to verify that you have nodes ready in your Kubernetes service

    ```bash
    bubectl get nodes
    ```
You should see a response with 1 node with status `ready` as a response.

4. Now you are ready to deploy the application to AKS.  First, you need to create a `deployment.yaml` file under the `aks_demo_all` folder and the content should be:

    ```yaml
    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: aks-demo
    spec:
      replicas: 2
      selector:
        matchLabels:
          app: aks-demo
      template:
        metadata:
          labels:
            app: aks-demo
        spec:
          containers:
          - name: aks-demo
            image: tailspinacr1000.azurecr.io/aks-demo:1.0
            ports:
            - containerPort: 3000
    ```

5. Issue the `kubectl` command to apply the deployment file

    ```bash
    kubectl apply -f deployment.yaml
    ```

6. Check for the pods created as you should see 2 pods based on the deployment.yaml file requesting 2 replicas

    ```bash
    kubectl get pods
    ```

![AKS Get Pods](../../Hands-on%20lab/images/aks_pods.png)

7. Now it is time to create a new `service.yaml` file in the same folder to expose the application and set the load balancer.

    ```yaml
    apiVersion: v1
    kind: Service
    metadata:
      name: aks-demo-service
    spec:
      type: LoadBalancer
      selector:
        app: aks-demo
      ports:
        - port: 80
          targetPort: 3000
    ```

8. Apply the `service.yaml` file

    ```bash
    kubectl apply -f service.yaml
    ```

9. Execute the following command to get the EXTERNAL IP Address for the service

    ```bash
    kubectl get svc aks-demo-service
    ```

The result will contain the IP Address of the `EXTERNAL-IP`. You might have to wait few minutes to get that IP Address.

![AKS External IP](../../Hands-on%20lab/images/aks_external_ip.png)

10. Verify that all is working by navigating to the url http://<EXTERNAL-IP> in your browser.

11. You have the ability to scale up live by using the following command and verifying the replicas

    ```bash
    kubectl scale deployment aks-demo --replicas=5
    ```

This scale command will increase the instances from 2 to 5 and add them to the Load Balancer automatically.

12. To verify, execute the following command

    ```bash
    kubectl get pods
    ```

![AKS Scale](../../Hands-on%20lab/images/aks_scale.png)

> **Important:** A very common and important scenario is what changes need to take place when the workload is an application that uses Azure SQL Databases or Azure SQL MI instances. The answer here is that AKS deployment DOES NOT fundamentally change just because the application uses Azure SQL Database or SQL MI. What does change are configuration, security, and connectivity considerations, not Kubernetes itself.

### What Does NOT Change
These steps remain identical:

|Area	| Status|
|---|---|
|Dockerfile	| Same|
|Image build | Same|
|Push to ACR | Same|
|AKS cluster creation | Same|
|Deployment object | Same|
|Service / Ingress | Same|
|Scaling | Same|
 
 Kubernetes does not care what database you use.

### What DOES Change When Using SQL Server
There are 4 key areas that must be addressed professionally:
1.	Connection string management  - modify the `deployment.yaml` file to include the server, database, username and password loaded ferom kubectl secrets
2.	Secrets handling - use the `kubectl create secret generic sql-credentials...`
3.	Network connectivity - most important part: allow AKS outbound IP or VNet, private endpoint (preferred)
4.	Identity & security - Can be done with SQL Authentication but Entra ID is preferred so Managed Identify is the recommended authentication method for AKS with SQL based apps.
